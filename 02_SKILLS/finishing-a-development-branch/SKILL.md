# 개발 브랜치 마무리하기

## 개요

명확한 선택지를 제시하고, 사용자가 고른 워크플로를 처리하면서 개발 작업의 마무리를 돕는다.

**핵심 원칙:** 테스트 검증 -> 선택지 제시 -> 선택 실행 -> 정리.

**시작할 때 알리기:** "이 작업을 마무리하기 위해 finishing-a-development-branch 스킬을 사용하겠습니다."

## 절차

### 1단계: 테스트 검증

**선택지를 제시하기 전에, 반드시 테스트가 통과하는지 확인한다:**

```bash
# 프로젝트 테스트 스위트 실행
npm test / cargo test / pytest / go test ./...
```

**테스트가 실패하면:**
```
테스트 실패 (<N> failures). 완료 처리 전에 먼저 수정해야 합니다:

[Show failures]

테스트가 통과하기 전에는 merge/PR을 진행할 수 없습니다.
```

중단한다. 2단계로 진행하지 않는다.

**테스트가 통과하면:** 2단계로 진행한다.

### 2단계: 기준 브랜치 확인

```bash
# 일반적인 기준 브랜치 확인 시도
git merge-base HEAD main 2>/dev/null || git merge-base HEAD master 2>/dev/null
```

또는 이렇게 묻는다: "이 브랜치는 main에서 갈라졌나요? 맞나요?"

### 3단계: 선택지 제시

정확히 다음 4가지 선택지를 제시한다:

```
구현이 완료되었습니다. 어떻게 진행할까요?

1. <base-branch>에 로컬로 병합
2. 푸시하고 Pull Request 생성
3. 브랜치를 현재 상태로 유지 (나중에 직접 처리)
4. 이 작업 폐기

어떤 옵션을 원하시나요?
```

**설명을 덧붙이지 말 것** - 선택지는 간결하게 유지한다.

### 4단계: 선택 실행

#### 옵션 1: 로컬 병합

```bash
# 기준 브랜치로 전환
git checkout <base-branch>

# 최신 내용 가져오기
git pull

# 기능 브랜치 병합
git merge <feature-branch>

# 병합 결과에서 테스트 검증
<test command>

# 테스트가 통과하면
git branch -d <feature-branch>
```

#### 옵션 2: 푸시하고 PR 생성

```bash
# 브랜치 푸시
git push -u origin <feature-branch>

# PR 생성
gh pr create --title "<title>" --body "$(cat <<'EOF'
## 요약
<2-3 bullets of what changed>

## 테스트 계획
- [ ] <verification steps>
EOF
)"
```

#### 옵션 3: 현재 상태 유지

보고: "브랜치 <name>을(를) 유지합니다. 현재 상태는 그대로 보존됩니다."

#### 옵션 4: 폐기

**먼저 확인한다:**
```
다음 항목이 영구적으로 삭제됩니다:
- 브랜치 <name>
- 모든 커밋: <commit-list>

확인하려면 'discard'를 입력하세요.
```

정확한 확인 입력을 기다린다.

확인되면:
```bash
git checkout <base-branch>
git branch -D <feature-branch>
```

## 빠른 참고표

| 옵션 | 병합 | 푸시 | 브랜치 유지 | 브랜치 정리 |
|--------|-------|------|-------------|----------------|
| 1. 로컬 병합 | ✓ | - | - | ✓ |
| 2. PR 생성 | - | ✓ | ✓ | - |
| 3. 현재 상태 유지 | - | - | ✓ | - |
| 4. 폐기 | - | - | - | ✓ (강제) |

## 자주 하는 실수

**테스트 검증 건너뛰기**
- **문제:** 깨진 코드를 병합하거나, 실패하는 PR을 만들게 됨
- **해결:** 선택지를 제시하기 전에 항상 테스트를 검증한다

**열린 질문으로 묻기**
- **문제:** "다음엔 뭘 해야 할까요?" -> 모호함
- **해결:** 구조화된 4가지 선택지를 정확히 제시한다

**폐기 시 확인 없음**
- **문제:** 작업을 실수로 삭제할 수 있음
- **해결:** 반드시 "discard"를 직접 입력해 확인받는다

## 위험 신호

**절대 하지 말 것:**
- 테스트가 실패하는데 진행하기
- 병합 결과의 테스트를 검증하지 않고 병합하기
- 확인 없이 작업 삭제하기
- 명시적 요청 없이 강제 푸시하기

**항상 할 것:**
- 선택지를 제시하기 전에 테스트 검증하기
- 정확히 4가지 선택지 제시하기
- 옵션 4에서는 직접 입력한 확인 받기
